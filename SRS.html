1.1 Purpose

The purpose of this Software Requirements Specification (SRS) is to define the functional and non-functional requirements for the Minimum Viable Product (MVP) of the Youth Connect platform. This document will serve as the guiding reference for the initial development phase. Youth Connect (MVP) is a web platform designed to connect entrepreneurs with verified investors through video pitches and secure direct messaging, focusing on speed, simplicity, and core functionality.

1.2 Scope

The scope of the Youth Connect MVP is to build and launch a core platform that validates the primary use case: connecting entrepreneurs and investors via video.

In Scope (MVP Core Features):
  • Simple, role-based user registration with profile completion indicators.
  • Video pitch upload and management for Entrepreneurs, including status updates.
  • A simple, admin-managed verification system for Investors.
  • An interactive discovery page for Investors with Like, Watchlist, and Connection features.
  • A private connection and communication system.
  • A simplified meeting scheduler with reminders.
  • A basic admin panel for user and verification management.
Out of Scope (Future Enhancements):
  • Dual-role or role-switching functionality.
  • AI-powered legal document and pitch deck generators.
  • Advanced recommendation engines or analytics dashboards.
  • Public comments, group chats, or file-sharing capabilities.
  • In-app video calling or payment processing.

2.1 Product Perspective

Youth Connect (MVP) is a cloud-native web platform focused on a simple, three-step user journey: 1) Entrepreneurs upload video pitches, 2) Investors get verified and browse those pitches, and 3) Both parties connect via direct messaging and scheduling. The platform prioritizes a clean, uncluttered experience to facilitate meaningful connections quickly.

2.3 Constraints

  • Regulatory: Must comply with India's DPDP Act, 2023 for data handling.
  • Platform: The application will be a web-based platform only. Native mobile apps are not in scope for the MVP.
  • Simplicity: The primary constraint is to avoid feature creep and adhere strictly to the defined MVP scope to ensure a fast launch.

list_altSpecific Requirements

3.1 Functional Requirements

3.1.1 User Management

  • The system shall provide a single registration page where a user must select a fixed role: "Entrepreneur" or "Investor".
  • All new users shall be required to verify their email address before logging in.
  • The system shall provide a secure login mechanism using email and password.
  • Users shall be able to create and edit a basic profile (name, bio, profile picture, location).
  • A user's role, once selected at registration, cannot be changed.
  • The user dashboard shall display a profile completion indicator (e.g., a percentage bar) to encourage users to fill out their profiles.

3.1.2 Video Pitch System

  • Entrepreneurs shall have a dashboard to upload, view, edit, and delete their video pitches.
  • The video upload form shall accept a single video file (e.g., MP4 format, max size 100MB, max duration 3 minutes).
  • The form shall include fields for: Business Name, Industry (dropdown), Funding Amount Sought, and a Short Description (max 500 characters).
  • Entrepreneurs shall be able to set a status for their pitch from a predefined list (e.g., "Actively Seeking", "In Talks", "Closed").
  • Uploaded pitches shall only be visible to verified Investors and Admins.

3.2 Non-Functional Requirements

3.2.1 Performance Requirements

  • The system shall support a minimum of 50 concurrent users.
  • API response times shall be under 800ms on average.
  • The platform's core pages shall load in under 5 seconds.

3.2.2 Security Requirements

  • The system shall use JWT-based authentication for all user sessions.
  • All data in transit shall be encrypted using HTTPS.
  • All user passwords shall be securely hashed and salted in the database.
  • The system shall implement basic protections against the OWASP Top 10, including input validation to prevent XSS and parameterization to prevent SQL Injection.

3.2.3 Usability Requirements

  • The user interface shall be responsive and functional on modern desktop and mobile web browsers.
  • Key user flows (uploading a pitch, verifying an account) must be simple and intuitive.

layersSystem Design and Architecture

5.1 Architectural Style

The Youth Connect MVP will be built using a Three-Tier Architecture, which separates the application into three logical and physical computing tiers: the presentation tier, the application tier, and the data tier. This model is well-suited for a web application of this scale, promoting modularity, scalability, and ease of development. The entire system will be cloud-native, designed for deployment on Amazon Web Services (AWS).

5.2 Presentation Layer (Frontend)

  • Framework: React.js. A modern, component-based JavaScript library will be used to build a dynamic, responsive, and intuitive user interface.
  • State Management: A global state management library (e.g., Redux Toolkit or Zustand) will be used to manage application-wide state, such as user authentication status and profile data.
  • Communication: The frontend will communicate with the backend exclusively through a secure REST API, sending and receiving data in JSON format. All API requests will be authenticated using JSON Web Tokens (JWT).

5.3 Application Layer (Backend)

  • Framework: Node.js with the Express.js framework. This provides a lightweight, efficient, and scalable foundation for building the REST API.
  • Responsibilities:
    • User Authentication & Authorization: Manages user registration, login, and session control using JWTs. Implements role-based access control (RBAC) to restrict features to specific user types (Entrepreneur, Investor, Admin).
    • Business Logic: Handles all functional requirements, including pitch management, investor verification logic, connection requests, and meeting scheduling.
    • API Endpoints: Exposes a well-defined set of RESTful endpoints for all frontend operations.
    • External Service Integration: Manages connections to third-party services like AWS S3 for file storage and an email service for notifications.

5.4 Data Layer

  • Relational Database: MySQL. A robust, open-source relational database will be used to store structured data like user profiles, pitch metadata, connections, and meeting information. Its support for JSON data types also provides flexibility.
  • Object Storage: Amazon S3 (Simple Storage Service). All large, unstructured files, including entrepreneur video pitches and investor verification documents, will be stored securely in an S3 bucket. The backend will manage secure, pre-signed URLs for uploads and downloads to ensure data privacy.
